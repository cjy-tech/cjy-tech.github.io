핵심 AWS 개념
What is Cloud Computing
인터넷을 통해 IT 리소스와 애플리케이션을 온디맨드(on-demand)로 제공하는 서비스를 말하며, 요금은 종량 과금제.
클라우드 환경은 '보안'에서도 장점이 있다. 현재 미국의 많은 기관(ex> CIA, NASA)에서 클라우드를 사용하고 있다.
온프레미스 환경을 구성한다면 공간, 인적자원, 장비구입, 물리적 보안, 예산 등 많은 것을 고려해야 한다.

온프레미스 환경 : 엔트프라이즈의 물리적 범위 내에 존재하는 것으로 보통 사내 서버실이나 회사 자체의 데이터 센터에 존재

What Amazon is Web Services?
Amazon Web Services(AWS)는 컴퓨팅파워, 데이터베이스스토리지, 콘텐츠전송, 기타 다양한 기능을 제공하는 보안 클라우드서비스 플랫폼이다.
AWS 클라우드는 컴퓨팅 파워(ex> VM기반 서버), 스토리지 옵션, 네트워킹 및 데이터베이스와 같은 광범위한 인프라서비스를 종량 과금제에 따라 지불하면 즉시 사용할 수 있는 온디맨드 유틸리티 형태로 제공한다.
범주별 AWS: 핵심서비스
컴퓨팅 : Amazon EC2, AWS ElasticBeanstalk, Auto Scaling, AWS Lambda, Amazon Elastic Container Registry, Amazon Elastic Container Service, Amazon Lightsail, AWS Batch
네트워킹 : Amazon VPC, Amazon Route 53, AWS Direct Connect, Elastic Load Balancing
스토리지 : Amazon S3, Amazon EBS, Amazon CloudFront, Amazon Clacier, Amazon Elastic File System, Storage Gateway, AWS Snowball, Amazon Snowmobile
데이터베이스 : Amazon RDS, Amazon DynamoDB, Amazon Redshift, AWS Database Migration Service(DMS), Amazon ElastiCache
범주별 AWS: 플랫폼 서비스

모바일 서비스 : AWS Mobile Hub, Amazon SNS, Amazon Cognito, AWS Device Farm, Amazon Mobile Analytics, AWS Mobile SDKs, Amazon Pinpoint
사물인터넷 : AWS IoT, AWS Greengrass

AWS와 같은 클라우드 컴퓨팅 환경을 사용하는 장점
자본 비용을 가변 비용으로 대체 : IDC구축/임대, 장비 구입과 같은 대규모 초기 투자라는 방식대신 필요시에 사용한 만큼만 비용을 내고 사용하면 된다. 선행투자의 위험이 줄어든다.
AWS와 같은 서비스 공급자가 규모의 경제를 이루어 사용자 요금을 낮게 유지할 수 있다.
필요한 인프라 용량이나 비용에 대한 추정 부담 감소 : 클라우드 기반 환경을 초기에 구성한 이후 필요에 따라서 확장/축소가 가능하다. 초기 투자를 작게 시작하여 실재 운영 과정을 거치며 조절하는 것이 가능하다. 온프레미스 환경에서도 불가능한 것은 아니지만 장비구입, 설치, 환경 구성에 드는 시간이 일반적으로 클라우드 환경에서보다 오래 걸린다.
속도와 민첩성 개선 : 클라우드 기반 환경을 사용하면 온프레미스에서 직접 환경을 구성하는 것보다 민첩하게 변경이 가능해진다. 장비 구매/설치/구성/테스트/운영이라는 과정의 일부가 사라지거나 수행에 드는 비용이 감소한다.
데이터 센터 운영 부담 감소 : 직접 데이터센터를 구축하거나 운영, 관리할 필요가 없어진다.
글로벌 서비스 대응력 향상 : 클라우드 기반 환경을 사용하여 빠르게 글로벌 서비스를 제공할 수 있다.

AWS 데이터 센터(Data Center)

보통 수천개의 서버와 장비로 구성된다.
모든 데이터 센터는 온라인 상태로 연결된다.
서로 다른 데이터 센터는 물리적으로 어느정도 떨어져 위치하도록 하여 자연재해와 같은 문제로 인해 동시에 여러 데이터 센터에 문제가 발생하는 것을 최대한 방지한다.
AWS 가용 영역(Availability Zone : AZ)

하나 이상의 데이터 센터로 구성된다.
같은 데이터 센터가 2개 이상의 AZ에 포함될 수 없다.
결함 분리 방식으로 설계되며, 고속 프라이빗 링크를 통해 다른 AZ와 연결된다.
AZ도 특정 AZ의 장애상황이 다른 AZ에 영향을 주거나 동시에 장애를 경험하지 않도록 하기 위해(예:지진같은 자연재해) 지리적으로 떨어진 곳에 위치하고 있다.
AZ는 사용자가 선택할 수 있다. 장애상황 대응을 위해 서로 다른 AZ에 복제구성을 하는 것을 권장한다.
AWS 리전(Region)

하나의 리전은 두 개 이상의 AZ로 이루어진다.
AWS는 전 세계에 여러개의 리전을 유지하고 있다.
리전간 데이터 복제를 지원한다.
리전간 통신은 AWS 백본 네트워크로 연결 인프라를 사용한다. 하지만 문제가 발생할 경우 일반 인터넷을 통한 통신이 이루어질 수 있기 때문에 리전간 통신의 경우 암호화 통신을 하는 것을 권장한다.
서비스 대상 사용자에게 가까운 리전을 선택하여 서비스 품질을 올릴 수 있다.

엣지 로케이션(Edge Location)과 리전 엣지 캐시

최종 사용자에게 콘텐츠를 짧은 지연시간으로 제공할 수 있도록 AWS는 24개국에 걸쳐 55개의 도시에서 107개의 상호 연결 위치(96개의 엣지로케이션 및 11개의 리전 엣지 캐시) 글로벌 네트워크를 사용한다.
엣지 로케이션은 북미, 유럽, 아시아, 호주 및 남미에 위치해 있으며 Amazon Route 53 및 Amazon CloudFront와 같은 AWS 서비스를 지원한다.
리전 엣지 캐시는 Amazon CloudFront와 함께 사용되며, 엣지 로케이션에 남겨두어도 될 정도로 자주 액세스하지 않는 컨텐츠가 있을 때 활용한다.
CloudFront 사용지 종단에 데이터가 없다면 오리진에서 가져와야 하는데, 이러한 오리진 접근 빈도를 줄이기 위하여, 리전 엣지 캐시가 CloudFront와 오리진 사이에 위치하여 응답성능을 높여주게 된다.

Amazon VPC(Virtual Private Cloud)
Amazon VPC는 클라우드에 호스팅 되는 가상 네트워크 환경이다.

사용자는 사용자의 AWS 계정 전용으로 VPC를 구성할 수 있으며, 기본적으로 다른 VPC와 논리적으로 격리된다.
각 VPC는 하나의 리전내에 유지되며 리전기반으로 동작한다.
같은 리전안의 다른 가용 영역(AZ)에 있는 리소스는 같은 VPC에 존재할 수 있다. 따라서 동일 VPC에 포함되는 서로 다른 AZ를 사용하여 HA 구성을 할 수 있다.

Amazon EC2(Elastic Compute Cloud)
Amazon EC2는 Amzon에서 제공하는 가상 컴퓨텅 환경(VM 인스턴스)이다.

다양한 서버 운영 체제를 지원한다.
인스턴스 선택 : CPU, Memory, Storage, Network, Graphic 에 대한 다양한 유형의 옵션을 제공한다.
여러 조합의 인스턴스 타입이 존재한다.
자체 서버 이미지를 생성, 저장 및 재사용이 가능하다(Amazon Machine Image: AMI)
필요에 따라 하나 이상의 인스턴스를 추가, 삭제, 시작, 중지할 수 있다.
보안 그룹을 사용하여 인스턴스에서 송수신되는 트래픽을 제어할 수 있다.
EC2 인스턴스의 instance storage는 인스턴스에 직접 연결되어 IOPS가 높지만 instance를 재시작하면 이전에 사용했던 instance storage를 사용하지 못한다.
만일 재시작이 되어도 데이터를 유지하고 싶다면 EBS를 붙여 사용하는 것을 권장한다.

Amazon S3(Simple Storage Service)
Amazon S3는 관리형 클라우드 스토리지 솔루션(Object Storage)이다.

하나의 객체는 최대 5TB 크기를 넘을 수 없다.
99.999999999%의 내구성을 제공한다. 아주 높은 데이터 보관 신뢰성을 제공한다.
데이터는 자동으로 복제되어 중복 저장된다.
자체적으로 파일의 체크섬을 비교하여 파일이 손상된 경우 손상된 파일을 대체복구한다.
객체 추가, 삭제시 알림이나 워크플로우 또는 스크립트를 트리거 할 수 있다.
전송 데이터와 저장 데이터는 모두 자동으로 암호화 될 수 있다.
S3 버킷(Bucket)

객체를 저장하는 논리적 컨테이너로 볼 수 있다. S3에서는 버킷에 저장되는 파일을 '객체'라고 부른다.
객체 내용의 부분 수정이 불가능하며 객체 전체를 수정하는 방식이된다.
하나의 계정에 하나 이상의 버킷을 만들 수 있으며, 버킷은 고유한 이름을 가져야 한다. 버킷의 이름 생성은 용도에 맞게 신경써서 해주는 것이 좋다.
버킷에 대한 접근제어가 가능하다.
버킷과 버킷의 객체에 대한 리전을 선택할 수 있다. 데이터의 복제본도 해당 리전안에서 만들어진다.
버킷을 생성하는 것으로는 과금이 안되며, 실재로 파일이 저장되어야 비용이 발생한다.

비과금 : Amazon S3로 수신, S3와 같은 리전의 CloudFront나 EC2로 송신하는 경우

Amazon EBS(Elastic Block Store)
Amazon EBS는 개별 스토리지 볼륨을 제공하며 EC2 인스턴스에 연결하여 사용할 수 있다. 인스턴스에 직접 연결되기 때문에 짧은 지연시간을 제공한다.

파일의 부분 수정이 가능하다. S3같은 경우 파일의 일부만 수정해도 전체가 수정된다.
EBS를 백업한다면 최초 생성하자 마자 스냅샷을 생성하고 주기적으로 백업하면 증분백업이 되기 때문에 백업시간을 줄일 수 있다.
생성 이후에 암호화 옵션을 설정하는 것은 불가능하다, 생성시점에 설정해야 한다.

볼륨은 S3로 백업할 수 있다.
Amazon EFS(Elastic File System)
Amazon EFS는 완전 관리형 클라우드 파일 스토리지이다.

공유 스토리지로 사용할 수 있다. 여러 AZ에서 공용으로 사용하는 것도 가능하다.
페타바이트 규모의 파일 저장이 가능하다.
용량 변경을 간단하게 할 수 있다.
네트워크 파일 시스템 버전 4.0 및 4.1(NFSv4) 프로토콜을 지원한다.

Amazon Glacier
Amazon Glacier는 보안, 내구성 및 저렴한 비용을 제공하도록 설계된 데이터 아카이브 서비스이다.

99.999999999%의 내구성을 제공한다. S3와 동일한 신뢰성을 보장한다.
저장소 잠금 기능을 제공하여 잠금이 활성한 Glacier 데이터는 수정할 수 없다. 예를 들어, 데이터 불변성에 대한 규정을 준수하며 보관하는 경우 사용하기 좋은 기능이다.
전송 데이터 및 저장 데이터에 SSL/TLS 암호화를 지원한다.
대량 데이터의 장기 보관에 적합하다. 예를 들어, 법적인 규정으로 특정 데이터를 3년 이상 보관해야 한다면 S3보다는 Glacier를 사용한 보관을 고려하는 것이 좋다.
데이터 보관 수명 주기 정책에 맞춰서 S3와 Glacier를 조합하여 운영하는 것을 권장한다.
Glacier에 보관된 데이터는 다시 가져오는데 시간이 들기 때문에 미리 요청을 해두는 것이 좋다.(물론 급행사용료를 지불하면 더 빠르게 받아볼 수 있다)
고속 검색 기능을 지원하나 데이터의 크기에 따라 비용이 달라진다.
Amazon RDS(Relational Database Services)
Amazon RDS는 관계형 데이터베이스 기능을 제공하는 관리형 서비스이다.

확장 가능 : 컴퓨팅, 스토리지
자동화된 데이터 중복성과 백업 지원
지원하는 데이터베이스 엔진 : Amazon Aurora, PostgreSQL, MySQL, MariaDB, Oracle, MS-SQL
Amazon RDS는 아래와 같은 기능이 필요한 경우 사용을 검토

복잡한 트랜잭션 또는 복잡한 쿼리기능
보통 및 빠른 쿼리/쓰기 속도 : 최대 30000IOPS(15000 read + 15000 write)
단일 작업자 노드/샤드 사용
높은 내구성
만일 대규모의 읽기/쓰기 속도(예: 150000 쓰기/초) 혹은 아주 큰 데이터 처리와 샤딩이 필요한 경우나 NoSQL 계열 저장소가 적절한 곳에는 Amazon RDS를 사용하는 것을 권장하지 않는다.
RDS의 동작 특성을 이해하고 사용 여부를 결정하도록 한다.

Amazon DynamoDB
Amazon DynamoDB는 완전관리형 NoSQL 데이터베이스 서비스이다.

크기와 관계없이 일관되게 10ms 미만의 응답성을 보장한다고 한다.
테이블 크기 및 처리량에 제한이 없다.
여러 리전별 복제와 업데이트가 이루어지는 글로벌 테이블을 구성할 때 편하다.
SSD 기반으로 실행된다.
온디맨드 백업을 사용해 테이블의 전체 백업을 생성할 수 있다.
복잡한 SQL이 아니라 간단한 key-value 형태로 데이터를 저장하는 용도에 적합하다. 예를 들어, 사용자의 세션 정보나 간단한 장바구니 정보를 처리하는데 적절하다.
복잡한 트랜잭션이나 JOIN과 같은 기능을 사용하는 경우는 권장하지 않는다.
사용자는 처리량에 대한 요청 설정을 지정하여 DynamoDB가 필요한 리소스를 할당한다.
DynamoDB는 3-replica로 동작한다.

데이터 수정 요청이 접수된 쪽은 바로 수정되지만 나머지 2개 복제본은 수정 반영에 약간의 지연시간이 발생한다.(대략 20ms이하?)
분산환경에서의 동기화 이슈 때문이다. 따라서 데이터의 일관성을 일부 포기하여 eventual consistency를 제공한다.
AWS IAM(Identity and Access Management)
AWS IAM은 AWS 서비스와 리소스에 대한 액세스 및 인증을 중앙에서 관리할 수 있도록 서비스를 제공한다.

IAM은 AWS 계정에서 추가 비용 없이 제공되는 기능이다.

IAM 정책을 통해 하나 이상의 권한으로 관리 정책을 정의할 수 있다.

일반적인 패턴은 2개의 서비스를 양쪽 AZ에 위치시키고 ELB를 통해 양쪽 서비스와 연결하는 것이다.

생성된 VPC와 VPC내 subnet은 생성 이후에 수정이 불가능하기 때문에 조심해야 한다. 이후에 수정하려면 일이 커진다.
VPC내 자원이 있는 경우 VPC 삭제가 안된다!!!
계정별 과금이 발생하기 때문에 비용관리 측면에서 복잡해질 수 있다. 이 경우 organization을 활용하는 것을 고려해보자.

10.0.0.0/16 = 10.0.0.0에서 10.0.255.255 까지의 모든 IP
16은 서브넷마스크에서 사용하는 1의 개수를 말한다. 앞쪽 첫번째 섹션부터 할당된다.
16으로 설정하면 앞에서부터 2개 섹션이 서브넷 구분을 위한 값이고, 이 값은 변경할 수 없으며 뒤에 2섹션만 IP주소 할당에 사용된다.

http://jodies.de/ipcalc 사이트 활용하여 IP 할당에 대한 계산을 해볼 수 있다.



AWS VPC는 /16과 /28 사이의 CIDR 범위를 사용할 수 있다.

서브넷은 CIDR 범위로 나누어진 네트워크의 세그먼트 또는 파티션을 말한다.
예를 들어, CIDR /22인 VPC는 총 1024개의 IP를 포함하며, 4개의 서브넷은 각각 251개의 IP를 포함한다.
각 서브넷 CIDR 블럭에서 처음 4개의 IP 주소와 마지막 IP 주소는 고객이 사용할 수 없으며 인스턴스에 할당할 수 없다.
예를 들어, CIDR 블럭이 10.0.0.0/24인 서브넷은 아래 5개의 IP주소는 예약되어 사용자가 쓸 수 없다.

10.0.0.0 : 네트워크 주소
10.0.0.1 : AWS에서 VPC 라우터용으로 예약
10.0.0.2 : AWS에서 Amazon 제공 DNS에 매핑하기 위해 예약
10.0.0.3 : 차후 사용을 위해 AWS에서 예약
10.0.0.255 : 네트워크 브로드캐스트 주소로 AWS는 VPC에서 브로드캐스트를 지원하지 않으므로 이 주소를 예약한다.
권장 사항 : 서브넷을 사용하여 인터넷 접근성을 정의하여 사용한다. 이를 통해 퍼블릭 리소스와 프라이빗 리소스간에 명확한 서브넷 수준의 격리를 정의할 수 있다.

퍼블릭 서브넷

프라이빗 서브넷
인터넷 게이트웨이에 대한 라우팅 테이블 항목이 없으며, 퍼블릭 인터넷에서 직접 액세스 할 수 없다.
일반적으로 '점프박스'(NAT/프록시/배스천 호스트)를 사용하여 제한된 아웃바운드 전용 퍼블릭 인터넷 액세스를 지원한다.
서브넷 구성 권장 사항:

AZ당 1개의 퍼블릭과 1개의 프라이빗 서브넷으로 시작한다.
퍼블릭 서브넷보다 프라이빗 서브넷에 더 많은 IP를 할당한다. 일반적으로 보안을 고려하여 구성하면 퍼블릭 서브넷에 배치해야하는 서비스나 컴포넌트가 생각보다 많지 않다.
데이터 스토어 인스턴스, 배치처리 인스턴스, 백엔드 인스턴스는 프라이빗 서브넷에 배치하는 것이 좋다.
작은 크기 보다는 더 큰 크기의 서브넷을 고려한다.(/24 이상)

보안 그룹

기본적으로 모든 수신 트래픽을 거부하며, TCP, UDP 및 ICMP 프로토콜을 기반으로 필터링할 수 있는 허용 규칙을 사용한다.

보안 그룹이 출발지로 지정될 수 있다.
특정 IP를 기준으로 출발지/목적지를 지정하는 경우, 동적으로 IP가 바뀌게 되면 관리에 어려움이있다.
따라서 보안 그룹을 기준으로 하면 해당 보안 그룹에 해당되는 출발지만 관리하며 된다.
인스턴스가 추가되는 경우 보안그룹에 할당되도록 관리해주면 된다.

고가용성이란 무엇인가?
고가용성(High Availability : HA)은 인적 개입 없이 시스템 혹은 서비스의 가동 중단 시간이 최소화 되도록 보장하는 것이다.
일반적으로 가용성은 100%에서 시스템 가동 중단 시간을 뺀 것을 말한다.
모든 문제 상황을 대응하기는 어려우나 가용성을 높이기 위한 여러가지 방법이 존재하며, 가용성을 높이는 작업은 비용의 증가와 서비스 구성의 복잡도가 발생할 수 있다.

SNS의 특정 토픽에 메시지 발행한다.
SNS에서 해당 토픽에 구독 등록된 Lambda 트리거한다.
Lambda의 함수가 호출되어 인스턴스의 스냅샷을 생성하도록 한하고 함수의 로그를 CloudWatch에 전송한다.
아래 코드는 AWS Lambda 함수의 예이다.

# Snap_and_Tag Lambda function
#
# This function is triggered when Auto Scaling launches a new instance.
# A snapshot of EBS volumes will be created and a tag will be added.

from __future__ import print_function

import json, boto3

def lambda_handler(event, context):
    print("Received event: " + json.dumps(event, indent=2))

    # Extract the EC2 instance ID from the Auto Scaling event notification
    message = event['Records'][0]['Sns']['Message']
    autoscalingInfo = json.loads(message)
    ec2InstanceId = autoscalingInfo['EC2InstanceId']

    # Snapshot all EBS volumes attached to the instance
    ec2 = boto3.resource('ec2')
    for v in ec2.volumes.filter(Filters=[{'Name': 'attachment.instance-id', 'Values': [ec2InstanceId]}]):
        description = 'Autosnap-%s-%s' % ( ec2InstanceId, v.volume_id )

        if v.create_snapshot(Description = description):
            print("\t\tSnapshot created with description [%s]" % description)

    # Add a tag to the EC2 instance: Key = Snapshots, Value = Created
    ec2 = boto3.client('ec2')
    response = ec2.create_tags(
        Resources=[ec2InstanceId],
        Tags=[{'Key': 'Snapshots', 'Value': 'Created'}]
    )
    print ("***Tag added to EC2 instance with id: " + ec2InstanceId)

    # Finished!
    return ec2InstanceId

느슨하게 연결된 인프라(Loosely Coupled Infrastructure)
인프라를 구성하는 컴포넌트들이 지나치게 직접 연결되어 구성되는 방식은 인프라의 확장이나 유지, 관리에 어려움이 증가할 수 있다.
또한 강결합(tightly coupled) 구성은 장애가 발생되는 경우 문제가 크게 퍼져나갈 수 있으며, 장애 대응/복구에 어려움이 증가한다.

예를 들어 여러대의 웹서버와 애플리케이션 서버로 구성되는 경우 여러대의 웹서버와 애플리케이션 서버 사이에 로드 밸런서를 두어 연결하는 방식이 인프라를 구성하는 컴포넌트를 느슨하게 구성하는 것이다.

느슨하게 연결된 인프라 구성
Loosely coupled 구성의 장점

구성 요소 하나의 변경이나 장애가 다른 구성 요소에 영향을 주는 것을 줄이고 상호 종속성을 줄일 수 있다.
확상성이 높아진다.
느슨하게 연결된 인프라 구성 방식

관리형 서비스와 서버리스 아키텍처에 대하여 고려한다.
메시지 대기열(message queue)을 애플리케이션 사이에 둔다.
정적 리소스의 경우 Amazon S3와 같은 외부 저장소를 활용한다.
사용자 인증이나 사용자 상태정보를 서버나 서비스에 자체적으로 저장하지 않고 관리형 AWS 서비스(예:DynamoDB)를 사용하여 저장한다.
마이크로 서비스(Micro Service) 형태로 서비스를 구성하는 것을 고려한다.
마이크로 서비스 아키텍처
마이크로 서비스 기반 아키텍처는 구성 요소를 독립된 기능으로 나누어 느슨하게 결합된 방식으로 연결하여 전체 서비스를 구성합니다.

기능을 개별 컴포넌트로 분할하여 구성 : 각 컴포넌트는 기능의 완결성을 이루는 것이 좋다.
컴포넌트별 테스트 : 작은 단위의 테스트를 진행하기 편해진다. 단, 전체 서비스에 대한 테스트는 구조의 복잡도가 올라가 오히려 테스트비용이 올라갈 수 있다.
컴포넌트별 변경 : 컴포넌트간의 연동 인터페이스가 바뀌는 경우가 아니라면 각 컴포넌트 별로 변경하는 작업이 다른 컴포넌트에 영향을 주지 않고 수행될 수 있다.
컴포넌트 확장 : 영향도를 최소화하여 컴포넌트별로 요구사항을 만족하기 위해 확장이나 축소작업을 할 수 있다.
컴포넌트별 기술 스택 선택 : 각 컴포넌트별 요구사항에 적합한 기술스택을 선택할 수 있다.
마이크로 서비스 구성시 유의해야 할 점

기존 서비스를 마이크로 서비스로 전화하기 전에 기술 스택이나 요구사항을 만족시킬 수 있는 준비가 되어 있어야 한다.
컴포넌트별로 적합한 기술 스택의 선택은 장점이 될 수 있으나 전체 서비스 관점에서는 기술 스택의 복잡도와 관리비용이 증가할 수 있다.
컴포넌트별로 로그가 발생하고 기능의 완결성은 컴포넌트간 연동으로 이루어지기 때문에 문제 발생시 관련 로그의 추적이 어려워질 수 있다.
서비스를 구성하는 전체 컴포넌트에 대하여 모니터링이 이루어져야 한다.

Amazon Simple Queue Service(SQS)
Amazon SQS는 완전관리형 메시지 대기열 서비스이다.

메시지 손실과 처리량을 신경쓰지 않고 메시지를 전송할 수 있다.
메시지 대기열
표준대기열 : 최소 1회 전송. 최선의 노력으로 순서화. 속도를 높이기 위한 경우에 사용한다.
FIFO 대기열 : 정확한 1회 처리(exactly-once) 보장, 제한된 처리량(초당 최대 300건의 전송, 수신, 삭제), 속도가 표준 대기열보다 느리다
비동기식(async) 처리에 활용할 수 있다.
완전관리형 서비스이기 때문에 확장성을 보장한다.
모든 메시지가 중복되어 저장되기 때문에 안정성을 보장한다.
당연하지만 SQS의 동작 특성을 이해하고 사용해야한다.
Amazon Simple Notification Service(SNS)
Amazon SNS는 알림을 설정 및 운영하고 구독 서비스를 통해 다른 애플리케이션에 알림을 전송할 수 있는 관전 관리형 서비스이다.

토픽(topic)기반 발생/구독(pub/sub) 모델로 동작한다.
다양한 형태의 발행(publishing)지원 : 이메일, HTTP/HTTPS, SMS, SQS, 모바일 푸시 메시징, Lambda 함수
HTTP/HTTPS전달 방식을 선택했을 경우 전송 실패에 대하여 재시도에 대한 설정을 할 수 있다.
1:N 방식으로 수신자가 여러명일 수 있다.
SQS와 같은 메시징 서비스로 볼 수 있다.

자주 액세스 하는 리소스는 Amazon CloudFront를 통해 제공하는 것을 고려한다.
빠른 컨텐츠 제공을 위해 S3와 CloudFront를 연결하여 서비스하는 것을 고려한다. CloudFront의 캐싱을 활용하여 응답성을 높일 수 있다.
비관계형 데이터라면 Amazon DynamoDB와 같은 NoSQL 데이터베이스에 저장하는 것을 고려한다.
관계형 데이터라면 Amazon RDS에 저장하는 것을 고려한다.
